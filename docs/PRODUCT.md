> 這份文件講**產品要往哪走、為什麼、缺什麼**。技術現況看 `docs/dev-status.md`，原始 PRD 看 `docs/MVP.md`。
> 撰寫日期：2026-07-05。這是一份策略觀點文件，會隨拍板更新。

## 一、一句話定位

**Hither 是一群人一起出遊時，讓「不走散、找得到彼此、行程對得上、帳算得清」的隨身指揮中心。**

願景是明確的：**出國自由行、家庭出遊在規劃與移動的每一刻，都會像打開地圖一樣自然地打開 Hither。** 地圖回答「這個地方在哪」，Hither 回答「我這群人現在在哪、要去哪、什麼時候到、誰付了錢」。

要達到「不可或缺」，判準只有一個：**它必須在一趟旅程的每一天、每一段都被打開，而不是只有走散那一刻。** 這是目前產品和願景之間最大的落差，本文件其餘部分都圍繞這件事展開。

## 二、現在已經做到什麼（核心賣點現況）

（對照 `dev-status.md`，這些是真功能，不是 stub）

| 能力 | 現況 | 對使用者的價值 |
|------|------|----------------|
| 即時成員地圖 | 已上線 | 一眼看到全隊每個人在哪，不用一直傳「你到哪了」 |
| 集合點 / 燈籠 | 已上線 | 隊長設點，全員即時收到方向、直線距離、預估步行時間 |
| 多站可重排行程 | 已上線 | 一整天的行程用可左右滑的 carousel 呈現，滑到哪站地圖鏡頭跟到哪站 |
| 分裂小隊（subgroups） | 已上線 | 大隊臨時拆成小隊各自行動，彼此仍看得見 |
| 推播 + Quick Commands | 已上線 | 隊長下指令、隊員回報需求，走到背景也收得到 |
| Live Activity / 動態島 | 已上線（需 Dev Build） | 不開 App 也能在鎖定畫面看到集合資訊 |
| 匿名快速加入 | 已上線 | 輸入暱稱 + group code 就能進，零註冊門檻 |

**現況的本質**：Hither 已經把「**一群人在同一時空移動、避免走散**」這個場景做得相當完整。這是強項，但也是天花板——它只覆蓋旅程中「大家聚在一起走動」的那幾個小時。

## 三、目標客群

### 核心客群（現在就該打）
1. **年輕人結伴出國自由行**（3–8 人）：日韓、東南亞、歐洲自由行。痛點密度最高、傳播力最強、對新工具接受度最高。**這是拉新引擎。**
2. **家庭出遊**（含長輩、小孩）：主題樂園、夜市、大型景區。痛點是「顧小孩怕走丟」「長輩走比較慢會脫隊」，情緒強度最高。**這是留存與口碑引擎。**

### 拓展客群（產品成熟後）
3. **校外教學 / 畢業旅行**：老師帶一大群學生，走散是安全責任。可走 B2B（學校採購）。
4. **旅行社領隊 / 導遊**：一團 20–40 人，領隊最怕點名少人。B2B2C，可品牌化。
5. **戶外 / 登山 / 路跑社團**：離線與安全需求極強。
6. **大型活動 / 展場 / 演唱會揪團**：臨時大群集合。

拓展客群的共同點：**人更多、走散代價更高、離線需求更強**。核心功能打磨好，往這些場景延伸幾乎不用改產品邏輯，只需要調規模與加管理權限。

## 四、為什麼還「不是不可或缺」——誠實的差距分析

### 4.1 最大問題：使用場景太窄（這是根本問題，不是功能問題）

地圖之所以不可或缺，是因為它覆蓋**旅程全生命週期**：出發前查、決定去哪、路上導航、到現場探索。Hither 目前只覆蓋「**大家聚在一起移動**」這一段——這一段一天可能只有幾小時，而且只有「真的走散」時才有急迫性。

> **只在痛的那一刻有用的工具，養不出打開習慣。** 使用者不會為了「以防萬一走散」而每天打開它。

要變成剛需，必須把觸點延伸到**出發前（規劃）**和**旅程中的每一天（花錢、約時間、找彼此）**。這是第五節所有提案的核心邏輯。

### 4.2 資安與隱私（這是紅線，不是待辦）

即時位置是最敏感的個資，而且 Hither 分享的是**未成年人與家人的即時位置**。目前的缺口：

- **group code 是唯一門檻**：6 碼英數，任何人拿到（或暴力猜）就能加入、看到全隊即時位置。**缺 code 有效期、缺加入審核、缺隊長踢人、缺速率限制。**
- **匿名帳號無問責**：無法追蹤惡意加入者、無法封鎖、無檢舉機制。
- **位置明文（僅 TLS in transit，非 E2E）**：MVP 明確寫「不做 E2E 加密」。對「分享小孩位置」的場景，這是法遵與信任風險（GDPR、各地兒少個資法）。
- **缺隱私控制**：沒有「暫停分享位置 / 幽靈模式」「離開後確實停止廣播並清除」的明確保證與 UI。
- **位置資料保留政策未定**：`member_locations` 存多久、誰能查、如何刪除。

> **判準**：家長要放心把小孩的即時位置交給你，資安就必須是**賣點級別**的投入，不能是 backlog。這條沒過，家庭客群打不開。

### 4.3 營運成本（決定商業模式能不能成立）

- **即時定位廣播是成本大戶**：成本隨「同時在線人數 × 更新頻率」線性成長。`member_locations` 的寫入與 Realtime 扇出，在大團（旅行社 40 人）與高頻更新下會快速放大。**需要建立「每個活躍 group 每小時成本」的模型**，並用自適應更新頻率（靜止時降頻）壓成本。
- **地圖搜尋 / geocoding 的 ToS 風險**：目前搜尋用 Photon / Nominatim 的 OSM 公開實例，**這些不允許商業級流量**，規模化必須自架或改用付費供應商（成本項）。
- **Android 的 Google Maps SDK 要計費**：上 Android 後多一筆地圖成本。
- APNs / FCM 推播本身免費，Supabase 匿名登入成本低——這兩塊不是問題。

> 好消息：核心成本項（定位廣播）可控且有優化路徑；壞消息：**上規模前一定要先算清楚單位經濟**，否則旅行社大團會把成本打穿。

### 4.4 客群規模與冷啟動（成長的結構性阻力）

- **iOS only**：Android 仍是 `MVP+`。家庭與校外教學場景 Android 佔比高，**少一半市場**。Android 是拓展 TAM 的剛性前提。
- **「全隊都要裝」的網路效應門檻**：只要一個成員沒裝，隊長就得回去用 LINE。這是採用的最大殺手。**需要極低摩擦的加入路徑**（連結一點就進、QR、甚至網頁版 follower 免安裝先看得到地圖，再引導安裝）。
- **無帳號 = 無法再行銷、無跨旅程留存**：旅程結束關係就斷了，下一趟得重來，也拿不到回訪。

### 4.5 使用方式

- 除了「聚在一起走動」沒有其他打開理由 → 沒有習慣迴路。
- 出發前沒有協作規劃的入口、旅程後沒有回顧的出口，**旅程的頭尾兩端完全是空的**——而這兩端正是養成習慣的地方。

## 五、要變成「剛需」，該補哪些核心賣點

這是本文件的重點，直接回答「還需要增加哪些核心賣點」。

**篩選原則**（呼應「不要 AR 那種花裡胡哨的東西」）：
- ✅ **高頻**：一趟旅程被用很多次，不是一次性。
- ✅ **有痛**：不做這件事會真的難受 / 有金錢或安全損失。
- ✅ **黏人**：用了就離不開，或牽涉多人 / 金錢而不敢隨便換工具。
- ❌ 排除「看起來很酷但一趟用不到兩次」的（AR 導航、勳章、熱力圖）。

以下依「剛需強度 × 對願景的槓桿」排序：

### 🥇 賣點一：出遊分帳 / 共同記帳（最強剛需，優先做）

**痛點**：4 個朋友去日本，永遠有一個人先付、最後結不清、傷感情。這是結伴旅遊**每一天、每一餐**都會發生的痛。

**為什麼是剛需**：
- **超高頻**：一天記好幾筆。旅程中打開次數會遠超「集合」功能。
- **黏人到離不開**：帳一旦開始記在 Hither，整趟就不敢換工具——**這是天然的鎖定效應**。
- **它自己就是安裝理由**：很多人會為了分帳而全隊裝 App（Splitwise 就是靠這點起家）。

**為什麼是 Hither 做而不是 Splitwise**：Hither 已經有「這群人 + 這趟行程 + 這些地點」的完整脈絡。「在哪家店花的錢」可以直接掛到地圖上的那個點——**這是純分帳工具做不到的整合**。

> 這一項單獨就可能把「幾小時才開一次」變成「一天開五次」。**建議列為下一個核心賣點的第一順位。**

### 🥈 賣點二：離線可用（願景的硬前提）

**痛點**：出國 = 沒網路 / 漫遊很貴 / 進地鐵山區就斷線。而**恰恰是斷線的時候最容易走散**。一個斷網就癱瘓的協作 App，在最需要它的時刻掛掉。

**為什麼是剛需**：「像地圖一樣不可或缺」這句話，**字面上就要求離線能用**——Google Maps 能離線，使用者的心理基準就是這條。目前 MVP 明確寫「不做離線地圖快取」，這與願景直接衝突。

**最小可行範圍**（不用一步到位）：
- 行程 / 集合點 / 分帳資料**本地快取**，離線可讀可記，回線同步。
- 離線地圖圖磚快取（先做「已規劃區域」預下載即可）。
- 斷線時用**藍牙 / 近距離**交換最後位置（同場景近距離走散的救命稻草，技術較重，可後排）。

### 🥉 賣點三：全行程協作規劃（把旅程「頭」補上，養成習慣）

**痛點**：出發前的行程規劃現在散在 Google Maps 收藏、Excel、群組聊天。沒有一個「全隊共編、出發後直接接著用」的地方。

**為什麼是剛需 + 高槓桿**：
- 這是**出發前一週就開始打開 App 的理由**——習慣迴路的起點。
- 現有的「多站可重排行程」已經是半成品底座，**只差把編輯權從「旅程中隊長」開放成「出發前全員共編 + 投票」**。
- 已規劃的 **Google My Maps KML 匯入**（見 `project_avatar_map_location` 記憶）正好是這塊的入口，把使用者既有的收藏一鍵搬進來，降低冷啟動。

**內容**：全員共編行程、對景點投票、拖曳排順序、標注「想去 / 必去」、自動接續到旅程當天的導覽。

### 賣點四：走散求救 + 家庭安全模式（情緒剛需，打開家庭客群）

**痛點**：「我在澀谷路口跟不上了」「小孩在樂園不見了」。這是**恐慌時刻**，情緒強度最高。

**為什麼是剛需**：安全是家庭客群的**購買理由**，不是加分項。

**內容**：
- 一鍵「**我走散了，快來找我**」→ 全隊（或隊長）收到高優先推播 + 我的最後位置 + 一鍵導航過來。走到背景、鎖屏都要收得到。
- **脫隊自動警示**：成員離開群體超過設定距離自動提醒隊長（家長）。
- **兒童 / 長輩安全模式**：被看護對象位置常駐、電量低提醒、抵達安全點通知。

> 這一項把 Hither 從「工具」升級成「守護」，是家庭場景的口碑核心。**但務必和 4.2 的資安一起做**——安全功能建立在不安全的資料層上，是負分。

### 賣點五：自由活動的「集合約定」（旅行團 / 自由行都剛需）

**痛點**：「現在 3 點，5 點在這個噴泉集合」——自由活動解散再集合是團體旅遊的標準動作，現在靠喊、靠記、總有人遲到走丟。

**內容**：集合點 + **集合時間**（現在只有地點沒時間）、倒數計時、「誰已到 / 誰還在路上 / 誰逾時」一覽、快到集合時間自動提醒。這是現有集合點功能的**自然延伸**，成本低、剛需明確。

## 六、優先順序建議（給拍板用）

| 優先 | 項目 | 理由 | 粗估投入 |
|:---:|------|------|:---:|
| **P0** | 資安與隱私底座（4.2） | 家庭客群的紅線；先於一切安全功能 | 中 |
| **P0** | 分帳 / 記帳（賣點一） | 最強剛需，最強鎖定，最省拉新成本 | 中 |
| **P1** | 集合時間約定（賣點五） | 現有功能的低成本延伸，剛需明確 | 小 |
| **P1** | 全行程協作規劃 + KML 匯入（賣點三） | 補旅程「頭」，養習慣，底座已在 | 中 |
| **P1** | 走散求救 + 家庭安全（賣點四） | 家庭口碑核心，需搭 P0 資安 | 中 |
| **P2** | 離線可用（賣點二） | 願景硬前提，但工程較重，分階段 | 大 |
| **P2** | Android（4.4） | 拓 TAM 的前提，工程量大 | 大 |
| **P2** | 免安裝加入 / 網頁 follower（4.4） | 破解冷啟動網路效應 | 中 |

> 邏輯：先用 **P0 分帳** 把打開頻率從「幾小時一次」拉到「一天多次」並建立鎖定，同時補**資安底座**解鎖家庭客群；P1 把旅程頭尾補上養成習慣；P2 打掉成長的結構性阻力（離線、Android、冷啟動）。

## 七、黏著度 / 留存機制

- **分帳的鎖定**：帳記在這裡就不敢換工具（見賣點一）。
- **帳號化（輕量）**：從純匿名進化到「可選綁定，保留旅程歷史」。有歷史才有回訪、才能再行銷。**不必強制註冊**，保留零門檻加入，但給願意留下的人一個回來的理由。
- **旅程回顧（旅程「尾」）**：一趟結束後自動生成「這趟去了哪、走了多少、花了多少、一起的人」——輕量、可分享，帶來社群傳播與下一趟的入口。
- **跨旅程的「常用同伴」**：記住常一起出遊的人，下次開團一鍵揪，降低每趟的冷啟動。

## 八、刻意先不做（守住焦點）

呼應「不要花裡胡哨」，以下明確延後或不做，避免範疇蔓延稀釋剛需：

- ❌ **AR 路徑導引**：一趟用不到兩次，炫技不解痛。
- ❌ **勳章 / 成就 / 遊戲化**：對「不走散、算清帳」的核心價值零貢獻。
- ❌ **社群動態牆 / 陌生人社交**：偏離「已認識的一群人一起出遊」定位。
- ❌ **AI 自動排行程**：在分帳、離線、安全這些剛需補齊前，這是分心項。
- ❌ **步行 step-by-step 導航**：地圖 App 已做得很好，Hither 不重造輪子，專注「群體協調」這個地圖不管的縫隙。

> 原則：Hither 只做**「一群已認識的人一起移動」時，地圖與聊天軟體都填不了的那條縫**——找彼此、對行程、約集合、算帳、保安全。其餘一律讓給既有工具。
